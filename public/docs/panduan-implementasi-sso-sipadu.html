SIPADU SSO Guide

Panduan Implementasi SSO SIPADU untuk Aplikasi Surrounding

Dokumen ini menjadi standar implementasi untuk tim programmer aplikasi internal Bank DP Taspen agar integrasi SSO berjalan konsisten, aman, dan mudah diaudit.

Dokumen kerja internal • Versi 1.0.0 • Tanggal 8 April 2026

1. Tujuan Dokumen

Panduan ini dipakai sebagai juknis implementasi SSO dari SIPADU ke aplikasi surrounding. Setiap tim aplikasi mengikuti pola yang sama agar integrasi tidak berbeda-beda antar sistem.

Tujuan Utama User login sekali di SIPADU lalu dapat membuka aplikasi surrounding tanpa memasukkan password lagi.
Identitas Bersama Kunci identitas antar aplikasi adalah employee_id.
Kontrol Lokal Role dan hak akses tetap dikelola oleh aplikasi masing-masing.

2. Prinsip Arsitektur

  • SIPADU berperan sebagai portal login pusat dan penerbit identitas.
  • Aplikasi surrounding berperan sebagai penerima token SSO dan pembuat session lokal.
  • SSO hanya mengotentikasi identitas user, bukan menggantikan mekanisme role aplikasi.
  • Semua aplikasi yang ingin menerima SSO harus melakukan penyesuaian integrasi.
Catatan penting: Aplikasi kritikal seperti core banking dapat tetap memakai mode launch only dan login mandiri. Dokumen ini fokus untuk aplikasi surrounding yang menerima SSO.

3. Standar Data User

Setiap aplikasi surrounding minimal harus memiliki data berikut pada tabel user:

Field Status Keterangan
employee_id Wajib Primary key identitas antar aplikasi. Harus sama persis dengan SIPADU.
name Wajib Nama display user di aplikasi.
email Wajib Data pendukung identitas dan komunikasi sistem.
is_active Wajib Penanda user masih boleh login ke aplikasi.
last_login_at Disarankan Audit login terakhir.
last_sso_login_at Disarankan Audit khusus login melalui SIPADU.
Wajib diperhatikan: jika employee_id menggunakan leading zero, nilainya tidak boleh berubah. Contoh 0075019 tidak boleh menjadi 75019.

4. Checklist Persiapan Aplikasi Surrounding

  1. Tambahkan kolom employee_id pada tabel user jika belum ada.
  2. Pastikan seluruh user existing sudah diisi employee_id yang valid.
  3. Tambahkan endpoint penerima SSO, misalnya /sso/login.
  4. Tambahkan service validasi token.
  5. Tambahkan konfigurasi issuer, audience, dan shared secret di environment aplikasi.
  6. Tambahkan log audit login SSO.
  7. Uji login sukses, expired token, wrong audience, wrong signature, dan user tidak ditemukan.

5. Perubahan Database

5.1 Migration User

<?php

Schema::table('users', function (Blueprint $table) {
    $table->string('employee_id')->nullable()->unique()->after('id');
    $table->boolean('is_active')->default(true)->after('email');
    $table->timestamp('last_sso_login_at')->nullable()->after('last_login_at');
});

5.2 Tabel Audit Token Terpakai

Untuk mencegah replay token, disarankan menambahkan tabel yang menyimpan jti token yang sudah dipakai.

<?php

Schema::create('used_sso_tokens', function (Blueprint $table) {
    $table->id();
    $table->uuid('token_jti')->unique();
    $table->string('employee_id');
    $table->timestamp('used_at');
    $table->timestamps();
});

6. Konfigurasi Environment

Tambahkan konfigurasi berikut pada file .env aplikasi surrounding:

SSO_ENABLED=true
SSO_ISSUER=sipadu
SSO_AUDIENCE=dmtl-audit
SSO_SHARED_SECRET=isi_secret_per_aplikasi
SSO_LOGIN_ROUTE=/sso/login

Tambahkan juga pada file config aplikasi, misalnya config/services.php:

'sso' => [
    'enabled' => env('SSO_ENABLED', false),
    'issuer' => env('SSO_ISSUER', 'sipadu'),
    'audience' => env('SSO_AUDIENCE'),
    'shared_secret' => env('SSO_SHARED_SECRET'),
],

7. Endpoint Penerima SSO

Tambahkan route berikut:

Route::get('/sso/login', [SsoLoginController::class, 'store'])
    ->name('sso.login');

Endpoint ini bertugas untuk:

  • menerima parameter sso_token dari SIPADU,
  • memvalidasi token,
  • mencocokkan user berdasarkan employee_id,
  • membuat session login Laravel,
  • mencatat audit log login SSO.

8. Format Payload Token

Payload minimal yang perlu dikirim oleh SIPADU:

{
  "iss": "sipadu",
  "aud": "dmtl-audit",
  "sub": "2907997",
  "iat": 1770000000,
  "exp": 1770000060,
  "jti": "uuid-token",
  "user": {
    "employee_id": "2907997",
    "email": "ferrianmfatichin@gmail.com",
    "name": "Ferrian Muhammad Fatichin"
  }
}

Arti field:

  • iss: penerbit token, harus SIPADU.
  • aud: nama aplikasi tujuan.
  • sub: subject atau user utama, umumnya sama dengan employee_id.
  • iat: waktu token dibuat.
  • exp: waktu token kedaluwarsa.
  • jti: ID token unik untuk replay protection.

9. Validasi Keamanan Token

Aplikasi surrounding wajib melakukan validasi berikut sebelum login user:

  1. Signature token valid.
  2. iss sama dengan issuer SIPADU.
  3. aud sama dengan aplikasi penerima.
  4. exp belum lewat.
  5. jti belum pernah dipakai.
  6. User dengan employee_id ditemukan dan masih aktif.
Jangan langsung login user tanpa validasi token. Semua poin di atas wajib dijalankan, terutama signature, audience, dan expiration.

10. Contoh Service Validasi Token

<?php

namespace App\Services;

use Illuminate\Support\Arr;
use Illuminate\Support\Carbon;
use Illuminate\Support\Facades\Config;

class SsoTokenService
{
    public function decode(string $token): array
    {
        [$encodedPayload, $signature] = explode('.', $token, 2);

        $expectedSignature = hash_hmac(
            'sha256',
            $encodedPayload,
            Config::get('services.sso.shared_secret')
        );

        if (! hash_equals($expectedSignature, $signature)) {
            abort(403, 'Signature token tidak valid.');
        }

        $payload = json_decode(base64_decode($encodedPayload), true);

        if (! is_array($payload)) {
            abort(403, 'Payload token tidak valid.');
        }

        if (Arr::get($payload, 'iss') !== Config::get('services.sso.issuer')) {
            abort(403, 'Issuer token tidak valid.');
        }

        if (Arr::get($payload, 'aud') !== Config::get('services.sso.audience')) {
            abort(403, 'Audience token tidak valid.');
        }

        if (Carbon::createFromTimestamp(Arr::get($payload, 'exp'))->isPast()) {
            abort(403, 'Token sudah expired.');
        }

        return $payload;
    }
}

11. Contoh Controller Login SSO

<?php

namespace App\Http\Controllers;

use App\Models\User;
use App\Services\SsoTokenService;
use Illuminate\Http\RedirectResponse;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Auth;

class SsoLoginController extends Controller
{
    public function store(Request $request, SsoTokenService $ssoTokenService): RedirectResponse
    {
        abort_unless(config('services.sso.enabled'), 404);

        $token = (string) $request->query('sso_token');
        abort_if($token === '', 403, 'Token SSO tidak ditemukan.');

        $payload = $ssoTokenService->decode($token);
        $employeeId = data_get($payload, 'user.employee_id');

        $user = User::query()
            ->where('employee_id', $employeeId)
            ->where('is_active', true)
            ->first();

        abort_if(! $user, 403, 'User tidak terdaftar atau tidak aktif.');

        Auth::login($user);
        $request->session()->regenerate();

        $user->forceFill([
            'last_login_at' => now(),
            'last_sso_login_at' => now(),
        ])->save();

        return redirect()->intended('/dashboard');
    }
}

12. Replay Protection

Replay protection mencegah satu token yang sama dipakai berulang kali. Mekanisme minimal:

  1. Ambil nilai jti dari token.
  2. Cek apakah jti sudah pernah tersimpan pada tabel used_sso_tokens.
  3. Jika sudah ada, tolak login.
  4. Jika belum ada, simpan setelah login berhasil.
$alreadyUsed = UsedSsoToken::where('token_jti', $payload['jti'])->exists();

abort_if($alreadyUsed, 403, 'Token sudah pernah digunakan.');

UsedSsoToken::create([
    'token_jti' => $payload['jti'],
    'employee_id' => $payload['user']['employee_id'],
    'used_at' => now(),
]);

13. Audit Log

Selain replay table, disarankan juga ada audit log untuk pelacakan login SSO.

Field Keterangan
user_id ID user lokal yang berhasil login.
employee_id ID karyawan dari payload.
token_jti ID token unik untuk korelasi audit.
logged_in_at Waktu login berhasil.
ip_address Alamat IP user saat login.
user_agent Informasi browser/perangkat.

14. Role dan Hak Akses

SSO hanya membuktikan bahwa user tersebut valid dari SIPADU. Hak akses di aplikasi surrounding tetap harus mengikuti role lokal.

  • SSO tidak otomatis menjadikan user sebagai admin.
  • Role seperti auditor, approver, reviewer, atau admin tetap disimpan pada aplikasi lokal.
  • Jika user berhasil login via SSO tetapi tidak punya role lokal, maka akses aplikasi harus dibatasi sesuai kebijakan sistem.

15. Strategi Implementasi Tim

15.1 Pembagian Tanggung Jawab

Tim Tanggung Jawab
Tim SIPADU Menyiapkan payload token, konfigurasi aplikasi, shared secret, dan URL launch SSO.
Tim Aplikasi Surrounding Menambah employee_id, endpoint SSO, validator token, auto-login, dan audit log.
Tim QA Menguji happy path, invalid signature, expired token, wrong audience, replay token, dan user tidak ditemukan.

15.2 Urutan Rollout

  1. Pilih satu aplikasi Laravel sebagai pilot.
  2. Implementasikan seluruh checklist dokumen ini.
  3. Lakukan testing bersama SIPADU dan QA.
  4. Setelah pola stabil, jadikan sebagai template untuk aplikasi lain.
  5. Rollout bertahap ke seluruh aplikasi surrounding.

16. Checklist Uji Coba

  1. User valid dengan employee_id yang cocok dapat login otomatis.
  2. User tidak ditemukan ditolak.
  3. User nonaktif ditolak.
  4. Token expired ditolak.
  5. Signature salah ditolak.
  6. Audience salah ditolak.
  7. Issuer salah ditolak.
  8. Token replay ditolak.
  9. Audit log tercatat setelah login berhasil.
  10. Role lokal tetap berjalan normal setelah login via SSO.

17. Kesimpulan

Implementasi SSO SIPADU di aplikasi surrounding pada dasarnya memerlukan empat fondasi: employee_id sebagai identitas bersama, endpoint penerima token, validasi keamanan token, dan pembuatan session login lokal.

Jika semua tim mengikuti standar yang sama, rollout ke banyak aplikasi akan jauh lebih cepat, aman, dan mudah diaudit.
